# Stage C v1 SFT Dataset Specification

**Status:** Verified and approved for internal research training; public release review pending  
**Training-data policy:** Existing seven-source pool only  
**Release policy:** Research-only pending source-license review  
**Tokenizer:** Existing 16,000-token project tokenizer  
**Maximum sequence length:** 1,024

## Objective

Build a deterministic, response-only supervised instruction-fine-tuning dataset
for Stage C v1. This version will not add, scrape, or synthesize training examples.
It will correct the existing split, truncation, duplicate, integrity, and evaluation
contracts before any optimizer update is allowed.

## Locked source pool

| Source | Input allocation | Primary role |
|---|---:|---|
| Alpaca | 1,000 | General instruction retention |
| ChatDoctor | 1,000 | Medical dialogue |
| MedAlpaca | 1,000 | Medical instructions and flashcards |
| MedInstruct | 1,000 | Medical instruction tasks |
| MedMCQA | 1,000 | Multiple-choice medical QA |
| OpenMedInstruct | 1,000 | Mixed medical instructions |
| PubMedQA | 1,000 | Evidence-conditioned biomedical QA |
| **Total** | **7,000** | |

The prototype preparation accepted 6,999 records and rejected one malformed record.
It produced 1,370,130 supervised training tokens and 70,409 validation tokens, but
it had no sealed test split and must not be used for Stage C training.

No source may be silently added. Records may be removed for malformed structure,
duplicate identity/content, unresolved license policy, or inability to retain a
meaningful instruction inside 1,024 positions. The final manifest, not the planning
counts above, is authoritative.

## Canonical template

```text
Instruction:
{instruction}

Input or Options:
{optional context}

Response:
{target response}
```

The serialized token sequence is:

```text
<bos> + prompt + response + <eos> + padding
```

Labels use `-100` for BOS, instruction, context, separator, and padding. Only response
tokens and EOS are supervised. The SFT loss performs exactly one standard causal
shift between logits and these labels.

## Split contract

| Split | Target fraction | Use |
|---|---:|---|
| Train | 90% | Optimizer updates |
| Validation | 5% | Checkpoint selection and retention decisions |
| Test | 5% | One-time evaluation after selection |

The split seed is 42. Records are grouped globally by the SHA-256 of their normalized
instruction, context type, and context. A prompt group must exist in exactly one
split, including when its records came from different upstream sources. Duplicate
record IDs and normalized prompt-response pairs are rejected before splitting.

The final build must prove zero record-ID and prompt-group overlap for every split
pair. Test IDs and artifacts are hashed before training and test results may not
influence hyperparameters or checkpoint selection.

## Truncation contract

The previous prototype could discard the prompt when a response alone exceeded the
context window. Stage C v1 prohibits that behavior.

1. Preserve BOS, the complete instruction, and the `Response:` separator.
2. Preserve the full response when it fits after removing optional context.
3. Truncate optional context before truncating the response.
4. If necessary, truncate the response from the right.
5. Always retain at least one response token and EOS.
6. Reject an example when the instruction and separator leave no response capacity.

Context and response truncation are recorded separately by source and split.

## Integrity and provenance

The version-2 manifest must record:

- input corpus and tokenizer SHA-256 values;
- template version, maximum length, split seed, and fractions;
- accepted, rejected, and duplicate counts;
- accepted examples, licenses, and license decisions by source;
- split record-ID and prompt-group hashes;
- supervised, prompt, response, and sequence-length statistics;
- separate context/response truncation counts;
- byte sizes and SHA-256 values for every tensor and structured JSONL artifact;
- explicit split-overlap results.

The prepared arrays remain excluded from Git. The manifest and audit reports are the
portable evidence contract.

## Licensing gate

The existing metadata includes sources marked `review` or carrying noncommercial
terms. Stage C v1 remains research-only until the upstream terms and intended model-
weight distribution policy are reviewed and recorded. Dataset contents must never be
committed to this repository.

## Build and verification

```powershell
python scripts/sft/prepare_sft.py --config configs/sft_stage_c_v1.yaml
python scripts/sft/check_sft_dataset.py --root datasets/tokenized/sft_stage_c_v1
python scripts/sft/audit_stage_c_dataset.py
```

The builder refuses to replace a non-empty output directory. A rebuild therefore
requires an explicitly new versioned output path or deliberate removal of an
unpromoted failed build.

## Acceptance gates

- Exactly the seven locked sources are present.
- Every retained source has at least 900 accepted examples.
- Train, validation, and test are all non-empty.
- No record ID or prompt group crosses splits.
- Every example retains supervised response content and a supervised EOS.
- Instructions are never removed by truncation.
- Token IDs remain inside the existing vocabulary.
- Every generated artifact matches its manifest size and SHA-256.
- License status and unresolved restrictions are reported rather than hidden.
